# 单例模式

思路：一个类仅有一个实例，并提供一个访问它的全局访问点，共享

实现：不管尝试去创建多少次，它都只给你返回第一次所创建的那唯一的一个实例。构造函数需要能够判断自己是否创建过一个实例，如果没有那就创建并保存，如果有就拿来返回

Vuex中的单例设计模式：如果Vue已经有了store实例，并且实例与现在想要尝试注入的实例相同，就return，不注入了。如果没有或者尝试注入的实例不同，就赋值，然后注入

# 案例

使用单例模式实现一个模态框：

  • 写一个立即执行函数，相当于开辟一块独立的作用域。创建一个变量记录单例对象，然后返回一个方法，这个方法作为构造方法，内部判断单例对象是否存在如果不存在就设置属性，返回值就是这个单例对象。立即执行函数的返回值给一个变量MyF，现在MyF就是这个构造函数了。这就是闭包，利用闭包操作单例对象
  • 通过new MyF（）就能得到单例对象

使用单例模式实现唯一的Storage：

  • class写法：包括静态方法getInstance，实例方法getItem和setItem
  • 闭包写法：外部方法，内部顶一个storage变量作为单例对象，返回内部一个构造函数。因为单例对象还要定义两个方法，所以就需要先写一个构造函数作为单例对象的类。给这个函数的原型添加方法getItem和setItem

# 工厂模式

思路：将 new 操作单独封装，只对外提供相应接口。遇到new 时，就要考虑是否应该使用工厂模式

何为工厂：不暴露创建对象的具体逻辑，而是将创建对象的过程单独封装，那么它就可以被称为工厂。工厂模式的目的，就是为了实现无脑传参

使用场景：不想让某个子系统与较大的那个对象之间形成强耦合，而是想运行时从许多子系统中进行挑选

分类：

  • 简单工厂：传入参数生成对象
  • 工厂方法：提供一个创建产品的接口，由子类来确定具体要创建的对象。相当于提供构造函数，子类调用父类构造函数，然后子类有其他函数
  • 抽象工厂：父类是抽象类，有若干方法createXXX，子类重写父类方法
    • 父类如何实现抽象：createXXX方法内部只有一句：throw new Error（”父类不能直接调用方法“）
    • 子类继承抽象类实现：子类继承createXXX方法中调用其他构造函数获取实例，为具体产品类。具体产品类也都是继承自一个父类
    • 当子类不需要了，那就再创建一个子类

# 观察者模式

思路：一个发布者，一个订阅者。当某个状态发生变化，就触发了观察者的执行。好比如btn.addEventListener（"click", ()=>{}）

实现：

  • 发布者类：包括属性observers。包括方法addObserver、remove、notify（notify遍历this.observers去调用回调）
  • 订阅者类：包括方法回调

# 发布-订阅模式

思路：一个发布者，一个订阅者，一个第三方

注意点：需要防止内存泄露

例子：

  • 事件总线。eventBus是第三方。发布者调用emit，订阅者调用on

  • 如何实现事件总线：属性handlers记录事件和回调之间的关系。方法on、emit、off

    • on（eventName）：先判断有没有这个事件存在，如果没有就往handlers添加并设值为空数组。然后往数组里面添加回调
    • emit（eventName，...args）：先判断有没有这个事件存在，如果存在就调用它那个函数传入参数。handlers[eventName]（...args）
    • off（eventName）：splice

必须先订阅再发布吗：如果先发布，后订阅，应该先把这条消息保存下来，等到有对象来订阅它的时候，再重新把消息发布给订阅者。建立一个存放离线事件的堆栈，当事件发布的时候，如果此时还没有订阅者来订阅这个事件，我们暂时把发布事件的动作包裹在一个函数里，这些包装函数将被存入堆栈中，等到终于有对象来订阅此事件的时候，我们将遍历堆栈并且依次执行这些包装函数

# 装饰器模式

装饰器模式：在不改变原对象的基础上，通过对其进行包装扩展，使原有对象可以满足用户的更复杂需求。装饰类和被装饰类都只关心自身的核心业务，实现了解耦

实现：装饰器的类的构造器传入被装饰的类对象。然后this.xxx = xxx，然后方法里面调用被装饰对象的方法，和调用自己的方法
