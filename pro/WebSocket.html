# WebSocket是什么

是在HTML5之后出现的、基于TCP协议的一个网络协议。实现浏览器与服务器全双工通信，允许服务器主动发送信息给客户端。和HTTP协议基本没有关系，只是兼容了现有浏览器的握手规范，也就是借助了一部分HTTP协议

# 为什么需要WebSocket协议

  • HTTP协议有缺陷：只能由客户端发起通信

    HTTP协议如果想要实现【即使通讯】，就得由客户端不断地发送请求，也就是不断的建立HTTP连接，让服务器回复了原本不需要回复的次数。如果用WebSocket就只需要在有消息的时候再告诉客户端有消息，类似于触发回调

    HTTP例子：【A：有消息吗？B：没有。A：有消息吗？B：没有。A：有消息吗？B：没有。A：有消息吗？B：没有。A：有消息吗？B：没有】

  • HTTP协议的即时通讯，需要一直建立、断开连接，中间有时间间隔，所以会有延迟不算真正的即时

  • 不断发送HTTP请求耗费资源，因为HTTP协议是无状态的，所以每次建立HTTP连接都要传输身份鉴别信息，浪费服务器时间，浪费传输流量。而WebSocket只需要一次HTTP握手， 记录了身份鉴别信息，后续的WebSocket通信不需要再握手，避免反复解析身份鉴别信息

# 通信模型

  • 客户端创建socket连接向服务端发送请求。WebSocket握手请求中，比起HTTP协议的，多了：
    • 【Upgrade：websocket】、【Connection：Upgrade】，告诉服务器这是WebSocket协议的而不是HTTP，要找对应的助理
    • 【Sec-WebSocket-Key：xxx】：要验证服务器是不是WebSocket助理
    • 【Sec-WebSocket-Protocol：yyy】：同一个URL下不同服务需要的协议不同，告诉服务器要什么服务
    • 【Sec-WebSocket-Version】：选择版本
  • 服务器倾听socket等待接收连接请求。接收到请求后创建socket连接。返回头中也是包括【Upgrade：websocket】、【Connection：Upgrade】，告诉客户端，已经切换成为WebSocket协议。这一步之后就借用完了HTTP协议的内容，后续都是WebSocket自己的东西
  • 建立起了端对端的TCP连接，客户端和服务端可以通信

# 聊天室案例

WebSocket是让客户端和服务器全双工通信。要实现聊天室，是将所有客户端连接到服务器，服务器把收到的消息广播给所有客户端

  • 客户端：需要创建WebSocket实例ws，给ws进行事件监听：open、close、error、message
  • 服务端：需要获取WebSocket的Server实例server，给server进行事件监听：open、close、error、connection、message。在connection的回调中再设置message的回调

API与函数：

  • 前端：new WebSocket（'ws:localhost:8000'）、ws.onOpen、ws.onClose、ws.onMessage、ws.onError、ws.send（字符串）。在onMessage的e.data中可以获取后端发送来的消息
  • 后端：需要引入ws模块，new ws.Server（{port：8000}）、server.onOpen、server.onClose、server.onMessage、server.onError、server.onConnection、server.clients、client.send（）。onConnect的回调中再绑定onMessage。onMessage中可以获取到传过来的参数msg，然后用server.clients.forEach给每个建立连接的客户端用c.send（msg）广播消息

尝试发送图片：

  • 客户端上传图片，把图片用base64的格式，通过WebSocket发送给服务器：通过FileReader.prototype.readAsDataURL就能转为base64
  • 服务器把接收到的base64广播给客户端
  • 客户端对接收到的信息作类型判断，如果是base64就用于img的src标签

# socket.io库

搭建：配合express框架

  • 后端：引用express并创建app实例，引用http模块调用createServer(app)创建服务得到server。不能用app.listen，否则还是用http进行通信。引用socket.io的Server函数，Server(server)得到io实例，io就相当于ws库的后端的server，用于监听事件执行回调。给io绑定回调后，给server提供端口并且监听。给app设置一个响应，返回一个页面
  • 前端：在访问app会返回的页面中引入socket.io.js，然后调用io()得到socket，相当于前端的ws，可以堆io绑定事件执行回调

连接：当客户端访问这个服务端，会得到响应回来的界面，执行了const socket = io()这一句就算是连接成功

缺点：这种在app.req（）中返回一个前端页面的不是前后端分离。最好还是要前后端分离

前后端分离配合Vue3：

  • 后端：引用express创建app实例，引用http模块调用createServer(app)得到server。引用socket.io的Server，Server(server）得到io。io绑定connection事件和回调，给server提供端口，就能启动后端
  • 前端：使用vue3。安装vue-socket.io和socket.io-client两个模块。在main.js入口文件中引用，app.use（new VueSocketIO（{debug：true，connection（SocketIO（后端地址））}）），在需要用到socket的地方引入io并socket = io（）就行
  • 正常来说没问题，但还有个问题就是跨域。前端需要在vue.config.js中配置代理，不设置的时候可以发现会发送带有标志/socket.io的请求，把带有‘/socket.io’的请求代理到服务器的ip和port。后端也需要配置跨域，在创建io的时候，加上第二个参数配置对象，包括属性cors：{origin：*}
  • 正常来说没问题了，但是Vue3全局变量和Vue2的不同，不是Vue.prototype的形式，而是app.config.gloablProperties.xxx = yyy。需要找到vue-socket.io的模块包，找到dist目录下的vue-socketio.js，找到t.prototype.$socketio = this.io，t.prototype.$vueSocketIo = this，改成这句：t.config.globalProperties.$http=this.io,t.config.globalProperties.$vueSocketIo = this，解决Vue3没法用的问题（没办法，socket.io作者没有适配vue3）
