# 线程与进程

单进程浏览器时代：

  • 不稳定：插件是最容易出问题的模块，并且还运⾏在浏览器进程之中，所以⼀个插件的意外崩溃会引起整个浏览器的崩溃。一锅端。如果有多个标签页一个厨师也会影响其他
  • 不流畅：线程少，渲染模块、JavaScript执⾏环境以及插件都是运⾏在同⼀个线程中的

多进程浏览器改进：

  • 进程是相互隔离的，所以当一个页面或者插件崩溃时，影响到的仅仅是当前的页面进程或者插件进程，并不会影响到浏览器和其他页面
  • JavaScript也是运行在渲染进程中的，所以即使JavaScript阻塞了渲染进程，影响到的也只是当前的渲染页面，而并不会影响浏览器和其他页面，因为其他页面的脚本是运行在它们自己的渲染进程中的

浏览器的进程：

  • 浏览器主进程：1个进程。负责界面展示与交互、管理各个tab标签、管理和下载网络资源（也可以被单独分出来作为网络进程）
  • 渲染进程（内核进程）：每个tab页1个进程。负责html、css、js文件的解析、渲染、事件处理。如果两个⻚⾯属于同⼀站点，那这两个标签会使⽤同⼀个渲染进程
  • 插件进程：1个插件1个进程
  • GPU3d绘制进程：1个进程，负责3D绘制和硬件加速

渲染进程的五个线程：

  • GUI渲染线程：解析css和dom，构建DOM树和renderObject树。GUI渲染线程和JS引擎线程是互斥的，执行JS引擎线程的时候；GUI线程会被挂起。执行时机：当界面需要重绘（Repaint）或由于某种操作引发回流(reflow)时，该线程就会执行
  • JS引擎线程：处理Javascript脚本程序，例如V8引擎。单线程，每个Tab页什么时候都只有一个JS线程。一直等待着任务队列中任务的到来，然后加以处理。耗时JS可用webWorker去写，webWorker会创建一个子线程
  • 事件触发线程：用来控制事件循环。触发事件后把任务推入该线程，等待js线程执行完后再推入js线程的执行队列去等待js引擎处理
  • 定时器触发线程：js引擎是单线程，所以需要把定时任务单独抽出作为一个线程。计时完毕后，添加到事件队列中，等待JS引擎空闲后执行。
  • 异步HTTP请求线程：每一次请求都是单独开辟线程。如果设置有回调函数，异步线程就产生状态变更事件，将回调再放入任务队列中。因为这是单独的线程，所以异步请求不会被js线程阻塞，而js又是单线程的

进程通信：

  • 管道通信：在操作系统在内核中开辟的一段缓冲区。单向通信、依赖文件系统、只能血缘关系
  • 消息队列：用户可以在消息队列中添加消息、读取消息等。消息队列提供了一种从一个进程向另一个进程发送一个数据块的方法
  • 信号量

僵尸和孤儿进程：父进程退出了，而它的一个或多个进程还在运行，那这些子进程都会成为孤儿进程。子进程比父进程先结束，而父进程又没有释放子进程占用的资源，那么子进程的进程描述符仍然保存在系统中，称之为僵死进程

多个标签页的通信：找中介

  • websocket：服务端作为中介
  • localStorage

# 网页请求流程

  1. 【用户】输入统一资源定位符

  2. 【浏览器进程】接收URL，判断是不是url，如果不是就拼装成url，转发给【网络进程】

  3. 【网络进程】查看本地缓存

    • 有：读取本地缓存渲染页面
    • 没有：解析url，准备发送请求。解析url之后知道了协议也就知道了请求的服务器端口，知道了目标域名

  4. 解析域名。DNS解析较为耗时，可以考虑使用dns-prefetch优化

    • 搜索浏览器的缓存
    • 搜索操作系统的缓存
    • 搜索本地dns服务器的缓存
    • 搜索根dns服务器。如果没有，它会告诉本地 DNS 服务器，可以到域名服务器上继续查询，并给出域名服务器的地址
    • 搜索顶级域dns服务器，它告诉下一步怎么走
    • ……
    • 找到了之后会把ip地址返回给本地DNS服务器，本地DNS服务器缓存该条解析记录

  5. 建立安全连接

    • TCP三次握手：请求建立、允许建立、确认建立。Chrome有个机制，同⼀个域名同时最多只能建⽴6个TCP连接，如果在同⼀个域名下同时有10个请求发⽣，那么其中4个请求会进⼊排队等待状态，直⾄进⾏中的请求完成，所以不是说解析好了IP地址就马上能建立TCP连接

    • TLS四次握手得到密钥：

      先了解什么是对称加密和非对称加密

      • 对称加密：使用同样的规则（或者key）对数据进行加密。缺点：被第三方知道规则就出事了
      • 非对称加密：有加密密钥和解密密钥，解密密钥要藏起来，加密密钥要发给客户端

      然后了解什么是证书：服务端需要向CA申请SSL证书，证书里说明了域名属于谁、有效期、公钥与私钥

      然后是流程：

      1. clientHello。客户端发送一个随机数Client Random用于生成对称密钥、支持的密码套件（如RSA算法）、SSL/TLS版本

      2. ServerHello。服务器会确定自己所支持的SSL/TLS版本，如果都不支持，则直接关闭此次连接。发送内容包括：客户端发的密码套件中的一套，生成一个随机数Server Random，SSL/TLS版本。

        然后服务端还会发送自己的数字证书和公钥（CA用证书信息生成Hash值，用私钥将Hash值加密得到签名，把签名和证书一起发）

        最后告诉客户端自己发完了

        • 客户端收到响应后会对证书证实有效性。使用同样的Hash算法计算基本信息，得到该证书的Hash值。使用证书中CA的公钥，解密签名，得到Hash值H2。判断Hash1和Hash2是否相等

      3. ClientKeyExchange。再生成一个随机数pre-master预主密钥，并用刚才服务器发过来的加密密钥进行加密后发送，双方会根据这三个数生成密钥。然后客户端再发一个Change Cipher Spec告诉服务器现在开始使用之前商定好的加密算法进行加密通话。然后客户端发送握手摘要Encrypted HandShare Message表示客户端TLS协商已经ok

      4. ServerKeyExchange。和3差不多，发送Encrypted HandShare Message表示服务端TLS协商已经ok

      回顾：双方各持有一个随机数Client Random和Server Random。交换，假设都被拿到了。服务端发送加密密钥给客户端，客户端拿到了加密密钥，假设也被拿到了。客户端对数据进行发送，使用加密密钥进行加密，之后必须要解密密钥才能解开。因为解密密钥从来没有公开，所以第三方拿不到数据。而这个**“数据”**其实就是最终的密钥，后面就变成了对称加密

  6. 构建HTTP请求报文

    • 请求行
    • 请求头
    • 空行
    • 请求体

  7. 使用TLS的密钥进行HTTP报文加密

  8. 经过传输层、网络层、链路层、物理层的数据封装。加上源地址、目的地址等信息

  9. 经过路由器，解封，读取目的地址，封装，转发

  10. 如果遇到了CDN，并且CDN里面有请求的内容，就不需要继续了，没有就要继续，直到到达网关接入层，接着就到达域名所代表的服务器

  11. 然后会根据url的协议，找到默认的端口号，从而找到对应的进程

  12. 到达了对应的进程后，Nginx把数据报展开，得到HTTP报文，在报文的请求行中得到资源路径。根据Nginx的规则，得到目标路径host:protocol/path，然后转发数据报到对应的地址。如果后端使用了Nacos进行注册，会有网关根据/api判断属于哪个服务模块，然后转发给当前的服务模块的其中一个节点

  13. 访问这个节点（ip：端口号），会访问到服务端，发现是http请求，创建一个线程处理请求。

  14. Nginx处理响应报文然后发送。同样经过封装、解封、再封装、转发，直至到达浏览器的网络进程

  15. TCP四次挥⼿断开连接。如果浏览器或者服务器在HTTP头部加上Connection:Keep-Alive，TCP就⼀直保持连接。保持TCP连接可以省下下次需要建⽴连接的时间，提⽰资源加载速度

  16. 网络进程解析响应数据，将数据交给浏览器进程

    • 301/302：需要重定向，从Location⾃动中读取地址，重新发起http请求
    • 200，则继续处理请求。检查响应类型Content-Type，如果是字节流类型，则将该请求提交给下载管理器，该 导航流程结束，不再进⾏后续的渲染，如果是html则通知浏览器进程准备渲染进程准 备进⾏渲染

  17. 浏览器进程检查当前url是否和之前打开的渲染进程根域名是否相同，如果相同，则复⽤ 原来的进程，如果不同，则开启新的渲染进程

  18. 浏览器进程将数据交给渲染进程进行渲染

    • 根据HTML生成DOM结构树
    • 根据CSS生成CSS规则树
    • 根据两棵树得到渲染树（布局树），不可见的元素不会在里面
    • 根据渲染树计算每个节点的位置
    • 对布局树分层，得到图层树layer tree
    • 每个图层单独绘制
    • 合并图层

# 网页渲染

# 流程

一个渲染引擎主要包括：HTML解释器，css解析器，JavaScript引擎，布局Layout模块，绘图模块

渲染流程是逐步完成的，为了更好的用户体验，渲染引擎将会尽可能早的将内容呈现到屏幕上，并不会等到所有的html 都解析完成之后再去构建和布局 render 树。它是解析完一部分内容就渲染显示一部分内容，同时，可能还在通过网络下载其余内容

  1. 根据HTML生成DOM结构树

  2. 把css转换为浏览器内容理解的结构。转换样式表中的属性值，使其标准化例如换单位。计算出DOM树中每一个节点的具体样式，遵守 CSS 的继承和层叠两个规则

  3. 根据 DOM 树和 CSSOM 规则树构建渲染树。渲染树的节点被称为渲染对象，渲染对象是一个包含有颜色和大小等属性的矩形，渲染对象和 DOM 元素相对应，但这种对应关系不是一对一的，不可见的 DOM 元素不会被插入渲染树

  4. 当浏览器生成渲染树以后，就会根据渲染树来进行布局（也可以叫做回流）。为特定的节点生成专门的图层，并生成一棵对应的图层树。不是布局树中的每一个节点都包含一个图层，如果一个节点没有对应的图层，那么这个节点就从属于父节点的图层。满足一下两个条件中的任意一个，元素就可以被单独提升为一个图层

    • 拥有层叠上下文属性的元素
    • 需要剪裁的地方。如出现滚动条，滚动条也会被提升为单独的层

    然后渲染引擎将图层的绘制拆分成一个个绘制指令，比如先画背景、再描绘边框，形成一个列表

    渲染进程的主线程给合成线程发送绘制列表。合成线程生成图块。在首次合成图块时只采用一个低分辨率的图片，这样首屏展示的时候只是展示出低分辨率的图片，这个时候继续进行合成操作，当正常的图块内容绘制完毕后，会将当前低分辨率的图块内容替换。渲染进程维护了一个栅格化线程池，专门负责把图块转换为位图数据。合成线程会选择视口附近的图块，把它交给栅格化线程池生成位图，生成的位图给到合成线程，最后给浏览器进程。浏览器进程把页面内容绘制到内存，也就是生成了页面，然后把这部分内存发送给显卡，显卡接收到浏览器进程传来的页面后，会合成相应的图像，并将图像保存到后缓冲区，然后系统将前缓冲区和后缓冲区对换，向用户显示的就是前缓冲区的内容。

# 阻塞

js

  • js既会阻塞DOM的解析，也会阻塞CSS的解析，所以还会阻塞页面渲染，都是为了避免做无用功。js引擎线程和渲染线程是互斥线程
  • 阻塞后续js的执行。因为会有依赖关系，需要维护

css

  • style：由HTML解析器解析。不阻塞DOM的解析，但阻塞渲染。渲染就是生成render树，在这之前要dom树和css树，所以肯定都会阻塞渲染
  • link：由css解析器解析。不阻塞DOM的解析，但阻塞渲染。浏览器会派发一个新的线程(HTTP线程)去加载资源文件，得等下载完才会渲染。阻塞后面的js语句的执行（整个过程是：link下载，html解析器继续解析遇到script，就算js加载完成了也要等到link的样式先加载完成才能切换js线程执行。而因为script的阻塞，后续的js和DOM和CSS的解析也会被阻塞）
  • @import：会阻塞后续css的下载。必须等当前css被下载并解析后才能加载下一个css，说白了不能并行下载css。别用

# 优化

JS：

  • js放body的最后，body中间尽量不要写script标签。避免script标签阻塞html解析生成DOM树和CSS树构建
  • 使用async/defer去异步加载js，不导致GUI渲染线程阻塞
    • script：停止页面渲染，加载js，执行js完成后再继续渲染
    • async：不停止页面渲染，异步加载js，下载完成后立即执行
    • defer：不停止页面渲染，异步加载js，如果 DOM 树还没构建好，则先等 DOM 树解析好再执行

CSS：

  • css要放在顶部，它不会阻塞html的解析。更早的把css请求回来，渲染dom的时候一次就过。如果放在很下面，浏览器为了用户体验，会使用浏览器会默认样式，确保首次渲染的速度，但等css请求回来的时候又要重新构建CSSOM树，会重排和重绘。所以CSS一般写在headr中，让浏览器尽快发送请求去获取css样式

DOM树、CSSOM树：

  • 代码层级尽量不要太深
  • 减少CSS选择器的层级，因为选择器是从左向右进行解析的

减少重绘重排：

  • 操作DOM时，尽量在低层级的DOM节点进行操作
  • 不要使用table布局， 一个小的改动可能会使整个table进行重新布局
  • 使用CSS的表达式
  • 不要频繁操作元素的样式，对于静态页面，可以修改类名，而不是样式
  • 使用absolute或者fixed，使元素脱离文档流，这样他们发生变化就不会影响其他元素
  • 避免频繁操作DOM，可以创建一个文档片段documentFragment，在它上面应用所有DOM操作，最后再把它添加到文档中
  • 将元素先设置display: none，操作结束后再把它显示出来。因为在display属性为none的元素上进行的DOM操作不会引发回流和重绘
  • 将DOM的多个读操作（或者写操作）放在一起，而不是读写操作穿插着写。这得益于浏览器的渲染队列机制

# 缓存

# 浏览器缓存（http缓存）

缓存的目的：

  • 对于浏览器的缓存，主要针对的是前端的静态资源，最好的效果就是，在发起请求之后，拉取相应的静态资源，并保存在本地。大大的减少了请求的次数。减少了服务器的负担，提高了网站的性能
  • 加快了客户端网页的加载速度
  • 减少了多余网络数据传输

缓存的过程：

  1. 第一次加载资源，服务器返回 200，浏览器从服务器下载资源文件，并缓存资源文件与response header，以供下次加载时对比
  2. 下一次加载资源时，由于强制缓存优先级较高，先比较当前时间与上一次返回 200 时的时间差，如果没有超过 cache-control 设置的 max-age，则没有过期，并命中强缓存，直接从本地读取资源。如果浏览器不支持HTTP1.1，则使用 expires 头判断是否过期
  3. 如果资源已过期，则表明强制缓存没有被命中，则开始协商缓存，向服务器发送带有 If-None-Match 和 If-Modified-Since 的请求
    • 服务器收到请求后，优先根据 Etag 的值判断被请求的文件有没有做修改，Etag 值一致则没有修改，命中协商缓存，返回 304；如果不一致则有改动，直接返回新的资源文件带上新的 Etag 值并返回 200
    • 如果服务器收到的请求没有 Etag 值，则将 If-Modified-Since 和被请求文件的最后修改时间做比对，一致则命中协商缓存，返回 304；不一致则返回新的 last-modified 和文件并返回 200

缓存的位置：

  • Service Worker：最优先。不管是从 Memory Cache 中还是从网络请求中获取的数据，浏览器都会显示是从 Service Worker 中获取的内容
  • Memory Cache：内存缓存。一旦我们关闭 Tab 页面，内存中的缓存也就被释放了
  • Disk Cache：硬盘中的缓存，读取速度慢点，容量好，存储时间长。根据 HTTP Herder 中的字段判断哪些资源需要缓存，哪些资源可以不请求直接使用，哪些资源已经过期需要重新请求

强缓存与协商缓存：强缓存策略和协商缓存策略在缓存命中时都会直接使用本地的缓存副本，区别只在于协商缓存会向服务器发送一次请求

  • 强缓存：如果缓存资源有效，则直接使用缓存资源，不必再向服务器发起请求。可以通过两种方式来设置，分别是 http 头信息中的 Expires 属性和 Cache-Control 属性。后者优先级大于前者
    • Expires：服务器通过在响应头中添加 Expires 属性，来指定资源的过期时间。是一个绝对时间，它是服务器的时间，所以有可能不一致
    • Cache-Control：在 HTTP 1.1 中提出了一个新的头部属性就是 Cache-Control 属性。相对时间，并且都是与客户端时间比较，所以服务器与客户端时间偏差也不会导致问题。可以设置以下字段
      • public：可以被任何对象（包括：发送请求的客户端、代理服务器等等）缓存
      • private：只能被用户浏览器缓存，不允许任何代理服务器缓存，例如用于避免代理服务器CDN缓存
      • no-cache：需要先和服务端确认返回的资源是否发生了变化，如果资源未发生变化，则直接使用缓存好的资源，也就是说没有强缓存，但是会有协商缓存
      • no-store：禁止任何缓存，每次都会向服务端发起新的请求。不缓存
      • max-age：缓存的最大有效期，单位为秒
  • 协商缓存：命中协商缓存的前提：max-age过期了，并且有no-cache
    • Last-Modify/If-Modify-Since：第一次请求一个资源的时候，服务器返回的header中会加上Last-Modify，标识该资源的最后修改时间。再次请求该资源时，发送的请求头中会包含If-Modify-Since，该值为缓存之前返回的Last-Modify。服务器判断是否命中缓存
      • 如果命中缓存，则返回http304，并且不会返回资源内容，并且不会返回Last-Modify。都是对比的服务端时间。但单位只有到秒为止，不太准确。有可能会频繁的变化却没有重新加载，有可能会自动变化但其实内容不变却要重新加载
    • ETag/If-None-Match：返回的是一个校验码。保证每一个资源是唯一的，资源变化都会导致ETag变化。服务器根据浏览器上发送的If-None-Match值来判断是否命中缓存

关于F5、回车、Ctrl+F5：

  • F5：直接对本地的缓存文件过期，全部协商缓存处理，检查是否过期
  • Ctrl+F5：就当没有任何缓存全部请求
  • 回车：按照正常流程，本地检查是否过期，然后服务器检查新鲜度

# 浏览器本地存储

# cookie

第一次请求网页内容的时候没有任何cookie，服务器收到请求在响应里添加头部Set-Cookie，并且在Set-cookie里进行标识，在下一次请求的时候浏览器就会在HTTP请求里添加头部Cookie，并且用上Set-Cookie里的标识，这样服务器就可以给不同用户匹配不同的内容。主要用于：个性偏好设置、记录追踪用户行为。 是HTTP 协议无状态的一种补充和改良。可以设置生存周期、作用域。

  • 名称无法修改
  • 无法跨域。可以使用Nginx反向代理，或者往其他网站写Cookie。服务端的Session存储到一个节点，Cookie存储sessionId
  • Cookie的数量不能超过20个，每个Cookie的大小不能超过4kb
  • 在请求一个新的页面的时候都会被发送过去

有安全问题。在 JS 脚本里可以用 document.cookie 来读写 Cookie 数据，这就带来了安全隐患，有可能会导致“跨站脚本”（XSS）攻击窃取数据。

  • 属性“HttpOnly”会告诉浏览器，此 Cookie 只能通过浏览器 HTTP 协议传输，禁止其他方式访问
  • 另一个属性“SameSite”可以防范“跨站请求伪造”（CSRF）攻击，设置成“SameSite=Strict”可以严格限定 Cookie 不能随着跳转链接跨站发送。而“SameSite=Lax”则略宽松一点，允许 GET/HEAD 等安全方法，但禁止 POST 跨站发送
  • 还有一个属性叫“Secure”，表示这个 Cookie 仅能用 HTTPS 协议加密传输，明文的HTTP 协议会禁止发送

Cookie个性广告：广告商网站给你贴上 Cookie，访问网站的时候就能用Cookie读出你的身份，做行为分析，再推给你广告

字段：

  • Name：名称
  • Value：值，对于认证cookie，value值包括web服务器所提供的访问令牌
  • Size： 大小
  • Path：可以访问此cookie的页面路径。 比如domain是abc.com，path是/test，那么只有/test路径下的页面可以读取此cookie
  • Secure： 指定是否使用HTTPS安全协议发送Cookie。使用HTTPS安全协议，可以保护Cookie在浏览器和Web服务器间的传输过程中不被窃取和篡改。该方法也可用于Web站点的身份鉴别，即在HTTPS的连接建立阶段，浏览器会检查Web网站的SSL证书的有效性。但是基于兼容性的原因（比如某些网站使用自签署的证书）在检测到SSL证书无效时，浏览器并不会立即终止用户的连接请求，而是显示安全风险信息，用户仍可以选择继续访问该站点。
  • Domain：可以访问该cookie的域名，Cookie 机制并未遵循严格的同源策略，允许一个子域可以设置或获取其父域的 Cookie。当需要实现单点登录方案时，Cookie 的上述特性非常有用，然而也增加了 Cookie受攻击的危险，比如攻击者可以借此发动会话定置攻击。因而，浏览器禁止在 Domain 属性中设置.org、.com 等通用顶级域名、以及在国家及地区顶级域下注册的二级域名，以减小攻击发生的范围
  • HTTP： 该字段包含HTTPOnly 属性 ，该属性用来设置cookie能否通过脚本来访问，默认为空，即可以通过脚本访问。在客户端是不能通过js代码去设置一个httpOnly类型的cookie的，这种类型的cookie只能通过服务端来设置。该属性用于防止客户端脚本通过document.cookie属性访问Cookie，有助于保护Cookie不被跨站脚本攻击窃取或篡改。但是，HTTPOnly的应用仍存在局限性，一些浏览器可以阻止客户端脚本对Cookie的读操作，但允许写操作；此外大多数浏览器仍允许通过XMLHTTP对象读取HTTP响应中的Set-Cookie头。
  • Expires/Max-size ： 此cookie的超时时间。若设置其值为一个时间，那么当到达此时间后，此cookie失效。不设置的话默认值是Session，意思是cookie会和session一起失效。当浏览器关闭(不是浏览器标签页，而是整个浏览器) 后，此cookie失效

# session

特点：为了cookie的安全性，不能直接把用户密码之类的信息放在cookie。服务器会设置cookie，将Session ID加到Cookie中。如果黑客入侵拿到了Session ID ，因为Session ID也是没有规律的字符串，所以黑客无法依据Session ID 推断出用户的密码。其次，服务器在发送Cookie之前，会对Session ID进行签名，回句话说，当黑客修改了Session ID，Session ID 就会变成服务器识别不了的字符串

缺点：做了负载均衡后，假如一段时间内 A 的访问量激增，会转发到 B 进行访问，但是 B 服务器并没有存储 A 的 Session，会导致 Session 的失效

# jwt

  • 不需要记录与查询，只需要验证正确性。减轻了服务端压力
  • 当客户端是非浏览器平台时，cookie是不被支持的，此时采用token认证方式会简单很多
  • 无需考虑CSRF：由于不再依赖cookie，所以采用token认证方式不会发生CSRF，所以也就无需考虑CSRF的防御

# 安全

# XSS

分类：

  • 存储型。恶意脚本会存储在目标服务器上，当浏览器请求数据时，脚本从服务器传回并执行
  • 反射型。诱导用户访问一个带有恶意代码的 URL 后，用户浏览，浏览器解析带有 XSS 代码的数据后当做脚本执行。反射型 XSS 的恶意代码存在URL⾥
  • 基于DOM的，不涉及服务器。在Web资源传输过程或者在用户使用页面的过程中修改Web页面。属于前端漏洞，其他两个是服务端漏洞

防御：

千万不要相信任何用户的输入，对用户的输入进行转码或者过滤

  • 基于DOM的XSS攻击全部都是在浏览器端完成的，基于DOM的XSS攻击是属于前端的安全漏洞。无论是何种类型的XSS攻击，它们都是要首先在浏览器中注入恶意脚本，然后再通过恶意脚本将用户信息发送至黑客部署的恶意服务器上。所以要阻止XSS攻击，我们可以通过阻止恶意脚本的注入和恶意消息的发送来实现。服务器对输入的脚本进行过滤和转码，经过转码之后的内容，即使返回页面，脚本也不会执行
  • 对需要插入到 HTML 中的代码做好充分的转义。对于 DOM 型的攻击，主要是前端脚本的不可靠而造成的，对于数据获取渲染和字符串拼接的时候应该对可能出现的恶意代码情况进行判断

使用 CSP ，CSP 的本质是建立一个白名单，由服务器告诉浏览器哪些外部资源可以加载和执行，从而防止恶意代码的注入攻击

对一些敏感信息进行保护，比如 cookie 使用 http-only，使得脚本无法获取。也可以使用验证码，避免脚本伪装成用户执行一些操作

# CSRF

定义：冒充用户在站内的正常操作。攻击者诱导用户进入一个第三方网站，然后该网站向被攻击网站发送跨站请求。如果用户在被攻击网站中保存了登录状态，那么攻击者就可以利用这个登录状态，绕过后台的用户验证，冒充用户向服务器执行一些操作

常见方式：

  • GET 类型的 CSRF 攻击，比如在网站中的一个 img 标签里构建一个请求，当用户打开这个网站的时候就会自动发起请求请求img的src，带上cookie信息
  • POST 类型的 CSRF 攻击，比如构建一个表单，然后隐藏它，当用户进入页面时，自动提交这个表单
  • 链接类型的 CSRF 攻击，比如在 a 标签的 href 属性里构建一个请求，然后诱导用户去点击。载请附上原文出处链接及本声明

防御：

  • 同源检测。服务器根据 http 请求头中 **origin 或者 referer **信息来判断请求是否为允许访问的站点，从而对请求进行过滤。但是其实是可以伪造的，安全性差
  • token。通常第三方站点无法拿到这个 token, 因此也就是被服务器给拒绝
  • cookie认证。户访问网站页面时，向请求域名注入一个Cookie，内容为随机字符串，然后当用户再次向服务器发送请求的时候，从 cookie 中取出这个字符串，添加到 URL 参数中，然后服务器通过对 cookie 中的数据和参数中的数据进行比较，来进行验证。使用这种方式是利用了攻击者只能利用 cookie，但是不能访问获取 cookie 的特点。并且这种方法比 CSRF Token 的方法更加方便。如果网站存在 XSS 漏洞的，那么这种方式会失效
  • cookie设置SameSite为strict，值一共有SameSite、Lax、None。SameSite就是浏览器完全禁止第三方请求携带Cookie。Lax只能在Get请求。None的话请求会自动携带上 Cookie

# hash

实现方式：hash路由的实现方式：通过监听url中的hash变化来进行路由跳转

实现：

  • 初始化路由类。包括属性routes记录路由，currentUrl记录当前url，构造器中添加
  • 存储hash值和对应的回调，以及实现hash切换时执行回调
    • 创建一个方法添加hash和callback的绑定，callback||function(){}
    • 创建refresh方法，设置currentUrl为location.hash.slice(1) || '/'，然后执行这个currentUrl对应的callback。this.routes[this.currentUrl]()
  • 在构造的时候给 load和hashchange事件 都绑定refresh作为回调

使用：

  • 调用构造函数给window添加属性router
  • 给router添加设置hash和回调
  • 切换页面就是href跳转到#/，#/aaa，#/bbb

实现前进后退：

  • 创建一个数组history来储存过往的hash路由例如/blue，并且创建一个指针currentIndex来随着后退和前进功能移动来指向不同的hash路 this.currentIndex = this.history.length - 1，this.history = []
  • 写一个方法backOff
    • 判断currentIndex的值是否小于0，小于就变0，大于就 - 1
    • 然后修改location.hash，从hisotry数组里面根据下标取值
    • 然后调用访问那个hash的回调
  • 修改refresh：
    • 每次进入页面时，往history数组里添加当前hash，并且currentIndex +1，调用访问那个hash的回调
  • 解决问题：在页面进行回退的时候，因为还是会有路由的变化，所以还会多调用一次refresh函数，然后把currentIndex 再+1，history又添加一个值。所以在refresh进行判断，判断isBack。在constructor里面多加一个属性isBack = false，然后每次backOff的时候，就把this.isBack = true。然后在refresh判断，判断完之后记得把isBack重新改成false
