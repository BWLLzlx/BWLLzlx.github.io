# 版本

# 0

HTTP0.9：

  • 只有一个请求行，并没有HTTP 请求头和请求体，因为只需要一个请求行就可以完整表达客户端的需求
  • 服务器也没有返回头信息，这是因为服务器端并不需要告诉客户端太多信息，只需要返回数据就可以
  • 返回的文件内容是以ASCII 字符流来传输的，因为都是 HTML 格式的文件

# 1

HTTP1.0：有了新的需求而HTTP0.9不能满足，在浏览器中展示的不单是 HTML 文件了，还包括了 JavaScript、CSS、图片、音频、视频等不同类型的文件，需要支持多种类型的文件下载，文件格式不仅仅局限于 ASCII 编码，还有很多其他类型编码的文件。引入了请求头和请求体，以Key-Value 形式保存。先发请求行再发请求头，先发响应头再发响应行

  • 请求头的内容：包括请求的文件类型accept、请求的文件压缩格式accept-encoding、请求的文件语言accept-Charset、文件具体的编码格式accept-language
  • 响应头的内容：最终的压缩类型content-encoding、文件类型与编码格式content-type
  • 在相应行引入状态码：告诉服务器最终处理该请求的情况
  • 引入cache机制：包括有expires的强缓存、if-modified-since的协商缓存

HTTP1.1：

  • 改进持久连接：1.0每一次都要建立TCP请求再发送http请求。目前浏览器中对于同一个域名，默认允许同时建立 6 个 TCP 持久连接。但还是会有队头阻塞问题，虽然多个请求可以共用一个TCP管道，但是必须等前一个请求处理完成才能发送后面的请求
  • 添加host请求头：1.0是认为每台服务器都绑定一个ip地址一一对应。提供了虚拟主机的支持，服务器可以根据不同的 Host 值做不同的处理
  • 提供了更多的缓存策略：引入了ETag、if-match，if-none-match
  • 引入了range请求头，允许只传输资源的某一部分，返回码206。能断点续传
  • 引入了客户端 Cookie 机制和安全机制

# 2

影响 HTTP/1.1 效率的因素：TCP 的慢启动（这是TCP为了减少网络拥堵的策略，无法改变）、多条 TCP 连接竞争带宽、持久连接的队头阻塞、建立连接慢

  • 多条 TCP 连接竞争带宽：一个域名只使用一个 TCP 长连接，保证带宽不会被分
  • 队头阻塞：采用多路复用去解决，并行发送请求。多路复用是通过在协议栈中添加二进制分帧层来实现的。发生改变的只是传输方式
    • 浏览器准备好请求数据
    • 经过二进制分帧层处理，将请求分成一帧一帧的数据去传输，给每个帧一个请求ID
    • 经过TCP/IP协议传到服务器，服务器接收所有帧，合并成完整的请求
    • 服务器处理请求，响应行、响应头和响应体分别发送至二进制分帧层
    • 进制分帧层会将这些响应数据转换为一个个带有请求 ID 编号的帧，经过协议栈发送给浏览器
  • 建立连接慢：TCP三次握手要1.5个RTT，TLS四次握手要2个RTT

多路复用给HTTP2带来的其他特性：

  • header压缩
  • 请求优先级
  • 服务器推送：HTTP/2可以直接将数据提前推送到浏览器，当用户请求一个 HTML 页面之后，服务器知道该 HTML 页面会引用几个重要的 JavaScript 文件和 CSS 文件，那么在接收到 HTML 请求之后，附带将要使用的 CSS 文件和 JavaScript 文件一并发送给浏览器，这样当浏览器解析完 HTML 文件之后，就能直接拿到需要的 CSS 文件和 JavaScript 文件

HTTP2一定比HTTP1.1更好吗：

  • HTTP2的所有请求都是在同一个TCP管道上传输的，并且请求可以并行发送。但是一旦某一个HTTP请求丢包，会阻塞整个TCP连接

  • 虽然HTTP1.1的请求是串行发送的，但是他好歹有6个TCP连接，一个TCP连接阻塞不会影响其他的

丢包率越高，HTTP2的效率反而越差

# 3

思路：发明一个 TCP 和 UDP 之外的新的传输协议QUIC协议

  • 基于UDP，减少握手时间
  • 实现了多路复用：同一物理连接上可以有多个独立的逻辑数据流，解决队头阻塞
  • 在 UDP 的基础之上增加了一层来保证数据可靠性传输。提供了数据包重传、拥塞控制
  • 集成了 TLS

挑战：

  • 没有完整的支持
  • 系统内核对 UDP 的优化远远没有达到 TCP 的优化程度
  • 丢包率较高

# 响应码

# 1

临时响应，请求还在进行中。需要请求者继续执行操作

  • 100：让请求方继续请求。已收到请求的第一部分，正在等待其余部分
  • 101：确认并准备切换协议

# 2

成功处理了请求

  • 200：成功处理了请求
  • 201：已创建。请求成功并且服务器创建了新的资源
  • 202：已接受。但尚未处理
  • 203：非授权信息。服务器已成功处理了请求，但返回的信息可能来自另一来源
  • 204：无内容。服务器成功处理了请求，但没有返回任何内容
  • 205：重置内容。但没有返回任何内容，区别是要求客户端重置输入
  • 206：部分内容。服务器成功处理了部分 GET 请求

# 3

通常用来重定向

  • 300：针对请求的user agent，可执行多种操作，或提供操作列表供请求者选择
  • 301：永久移动。会自动将请求者转到新位置
  • 302：临时移动。老资源还在，只是临时给个新url，下次还得用旧地址。后面被细分为303和307，理论上302已经没用了，但是为了因为303和307是出现在HTTP1.1的，为了兼容所以还在使用。
    • 307用得少，只有用在POST请求重定向的场景
    • 303用得也少，因为以前的浏览器执行302就是303
  • 303：302本来在规范中是不允许重定向时改变请求方法的，往详细的说就是当客户端的POST请求收到302状态码的响应时需要先跟用户确认是否重发POST请求。但是很多浏览器却会重定向并改变请求方法，例如把Post改成Get，302规范并没有实现。在HTTP1.1把这种错误的处理作为一种新的规范303：POST重定向为GET
  • 304：请求的内容没过期，让客户端用缓存中的数据
  • 305：表示请求者应使用代理访问请求的网页
  • 307：做了302原本要求的事，也就是POST请求收到307时需要先跟用户确认是否重发POST请求

# 4

  • 400：请求的语法错误，服务器无法理解，例如传参传错、url含有非法字符
  • 401：请求要求身份验证。对于需要登录的网页，服务器可能返回此响应
  • 403：理解请求客户端的请求，但是拒绝请求
  • 404：无法根据客户端的请求找到资源
  • 405：禁用。禁用请求中指定的方法
  • 406：不接受
  • 407：需要代理身份验证。与401类似，但是需要使用代理
  • 408：请求超时
  • 409：冲突。服务器在完成请求时发生冲突
  • 410：请求的资源已永久删除
  • 411：不接受 不含有效内容长度标头字段的 请求
  • 412：未满足前提条件
  • 413：表示响应实在太大。服务器拒绝处理当前请求
  • 414：请求的 URI(通常为URL)过长，服务器无法处理
  • 415：请求格式不受请求页面的支持
  • 416：请求范围不符合要求
  • 417：预期内容无法被服务器满足
  • 418：发现客户端是一个爬虫
  • 420：方法失效
  • 422：请求含有语义错误，无法响应
  • 429：在给定时间内发送的请求太多
  • 499：客户端关闭请求

# 5

  • 500：服务器内部出错
  • 501：服务器不具备完成请求的功能
  • 502：错误网关
  • 503：服务器目前无法使用
  • 504：网关超时
  • 505：服务器不支持请求中所用的 HTTP 版本

# 请求方式

# GET

对应select操作，会把请求的参数附加在URL后面，这样是不安全的

# POST

对应update操作，请求参数都是请求body中

# HEAD

与GET方法类似，仅传输状态行和标题部分，用于获取报头

# OPTIONS

返回服务器针对特定资源所支持的HTTP请求方法，测试服务器的功能性

# PUT

对应insert操作

# DELETE

对应delete操作

# TRACE

回显服务器收到的请求，主要用于测试或诊断

# CONNECT

建立到给定URI标识的服务器的管道
